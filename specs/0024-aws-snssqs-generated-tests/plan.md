# Implementation Plan: AWS SNS/SQS Generated Tests

**Branch**: `minor.aws-snssqs.generated-tests` | **Date**: 2026-03-12 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `specs/0024-aws-snssqs-generated-tests/spec.md`

## Summary

Replace ~270 handwritten AWS SNS/SQS messaging gateway tests with generated tests using the existing Liquid template-based test generator framework. This requires: creating `test-configuration.json` files for both `AWS.Tests` and `AWS.V4.Tests` projects with four transport variants each (SNS Standard, SNS FIFO, SQS Standard, SQS FIFO), implementing 8 `IAmAMessageGatewayProvider` classes (4 per project), and validating against LocalStack via Podman.

## Technical Context

**Language/Version**: C# / .NET 10.0 (matches `BrighterTestTargetFrameworks`)
**Primary Dependencies**: xUnit, AWSSDK.SQS, AWSSDK.SimpleNotificationService, LocalStack, Podman
**Storage**: N/A (AWS resources are ephemeral test fixtures)
**Testing**: xUnit with `dotnet test`, LocalStack for AWS emulation
**Target Platform**: Cross-platform (Linux/macOS/Windows)
**Project Type**: Test infrastructure (tools + test projects)
**Constraints**: Must reuse existing Liquid templates without modifications; must be backward-compatible with existing credential chain

## Implementation Phases

### Phase 1: Configuration & Shared Infrastructure (MVP)

Create the `test-configuration.json` files and the shared `DefaultMessageBuilder` / `DefaultMessageAssertion` files (auto-generated by running the generator) for both AWS test projects.

**Tasks:**

#### Task 1.1: Create `test-configuration.json` for `Paramore.Brighter.AWS.Tests`

Create `tests/Paramore.Brighter.AWS.Tests/test-configuration.json` with the `MessagingGateways` dictionary containing all four transport variants. The configuration uses:
- **SnsStandard**: `SnsPublication` + `SqsSubscription`, no partition key, no delayed messages
- **SnsFifo**: `SnsPublication` + `SqsSubscription`, partition key enabled
- **SqsStandard**: `SqsPublication` + `SqsSubscription`, delayed messages enabled
- **SqsFifo**: `SqsPublication` + `SqsSubscription`, partition key + delayed messages enabled

All variants: no publish confirmation, DLQ enabled, broker validation enabled, requeue enabled.

**Reference**: `tests/Paramore.Brighter.RMQ.Async.Tests/test-configuration.json` (multi-gateway pattern)

#### Task 1.2: Create `test-configuration.json` for `Paramore.Brighter.AWS.V4.Tests`

Same structure as Task 1.1 but:
- Namespace: `Paramore.Brighter.AWS.V4.Tests`
- Publication/Subscription types use `Paramore.Brighter.MessagingGateway.AWSSQS.V4.*`
- Provider paths use `Paramore.Brighter.AWS.V4.Tests.MessagingGateway.*`

### Phase 2: SNS Standard Provider (Non-V4)

Implement the SNS Standard provider first as the simplest pub/sub variant. This validates the entire pattern before building the remaining three providers.

#### Task 2.1: Create `SnsStandardMessageGatewayProvider.cs`

File: `tests/Paramore.Brighter.AWS.Tests/MessagingGateway/SnsStandardMessageGatewayProvider.cs`

Implements both `IAmAMessageGatewayProactorProvider` and `IAmAMessageGatewayReactorProvider`.

**Key implementation details** (derived from existing handwritten tests):

- **`GetRoutingKey()`**: Generate unique routing key per instance using `$"sns-std-{Guid.NewGuid():N}"` (for topic name)
- **`GetChannelName()`**: Generate unique channel name `$"sns-std-ch-{Guid.NewGuid():N}"` (for queue name)
- **`CreatePublication()`**: Return `SnsPublication` with `TopicArn = null` (let infrastructure create it), `MakeChannels = OnMissingChannel.Create`
- **`CreateSubscription(RoutingKey, ChannelName)`**: Return `SqsSubscription` with `ChannelType.PubSub`, `MakeChannels = OnMissingChannel.Create`, `RequeueCount = 3`
- **`CreateProducerAsync(IAmAPublication)`**: Use `GatewayFactory.CreateFactory()` to get `AWSMessagingGatewayConnection`, create `SnsMessageProducer`
- **`CreateChannelAsync(IAmASubscription)`**: Use `ChannelFactory` to create async channel
- **`CreateProducer()` / `CreateChannel()`**: Sync variants using same factories
- **`CleanUp()` / `CleanUpAsync()`**: Delete topic and queue via `_producer.DisposeAsync()` and `_channel.Dispose()`
- **DLQ support**: Create a DLQ subscription with separate routing key, create a consumer for DLQ message retrieval
- **Requeue tracking**: Wrap channels with `RequeueTrackingChannelAsync`/`RequeueTrackingChannelSync` decorators (same pattern as Redis/RMQ providers)

**Reference files**:
- `tests/Paramore.Brighter.RMQ.Async.Tests/MessagingGateway/RmqClassicMessageGatewayProvider.cs` (overall structure)
- `tests/Paramore.Brighter.Redis.Tests/MessagingGateway/RedisMessageGatewayProvider.cs` (DLQ pattern)
- `tests/Paramore.Brighter.AWS.Tests/MessagingGateway/Sns/Standard/Proactor/When_posting_a_message_via_the_messaging_gateway_async.cs` (AWS-specific setup)

#### Task 2.2: Run generator and verify SNS Standard tests compile

```bash

dotnet run --project ../../tools/Paramore.Brighter.Test.Generator/Paramore.Brighter.Test.Generator.csproj
dotnet build
```

Verify the generated files appear under `MessagingGateway/SnsStandard/Generated/{Proactor,Reactor}/`.

### Phase 3: Remaining Three Providers (Non-V4)

#### Task 3.1: Create `SqsStandardMessageGatewayProvider.cs`

File: `tests/Paramore.Brighter.AWS.Tests/MessagingGateway/SqsStandardMessageGatewayProvider.cs`

Key differences from SNS Standard:
- **`CreatePublication()`**: Return `SqsPublication` (not `SnsPublication`)
- **`CreateSubscription()`**: Set `ChannelType.PointToPoint` (not `PubSub`)
- **`CreateProducerAsync()`**: Create `SqsMessageProducer` (not `SnsMessageProducer`)
- **Delayed messages**: Supported — no special provider code needed (template handles it)

**Reference**: `tests/Paramore.Brighter.AWS.Tests/MessagingGateway/Sqs/Standard/Proactor/When_posting_a_message_via_the_messaging_gateway_async.cs`

#### Task 3.2: Create `SnsFifoMessageGatewayProvider.cs`

File: `tests/Paramore.Brighter.AWS.Tests/MessagingGateway/SnsFifoMessageGatewayProvider.cs`

Key differences from SNS Standard:
- **`GetRoutingKey()`**: Append `.fifo` suffix: `$"sns-fifo-{Guid.NewGuid():N}.fifo"`
- **`GetChannelName()`**: Append `.fifo` suffix: `$"sns-fifo-ch-{Guid.NewGuid():N}.fifo"`
- **`CreatePublication()`**: Return `SnsPublication` with `SnsAttributes { Type = SnsType.Fifo }`
- **`CreateSubscription()`**: Return `SqsSubscription` with `SqsAttributes { Type = SqsType.Fifo }` and `FifoThroughputLimit.PerMessageGroupId`
- **Partition key support**: Enabled via config flag (template generates partition key test)

**Reference**: `tests/Paramore.Brighter.AWS.Tests/MessagingGateway/Sns/Fifo/Proactor/When_posting_a_message_via_the_messaging_gateway_async.cs`

#### Task 3.3: Create `SqsFifoMessageGatewayProvider.cs`

File: `tests/Paramore.Brighter.AWS.Tests/MessagingGateway/SqsFifoMessageGatewayProvider.cs`

Key differences from SQS Standard:
- **`.fifo` suffix** on routing keys and channel names
- **FIFO attributes** on both publication and subscription
- **Partition key**: Enabled
- **Delayed messages**: Enabled (SQS supports this for FIFO too)

**Reference**: `tests/Paramore.Brighter.AWS.Tests/MessagingGateway/Sqs/Fifo/Proactor/When_posting_a_message_via_the_messaging_gateway_async.cs`

## Risk Assessment

| Risk | Mitigation |
|------|-----------|
| Generated DLQ test may not match AWS redrive pattern (SQS uses visibility timeout, not explicit requeue) | The `RequeueTrackingChannel` decorator in the provider handles this — same pattern as Redis/RMQ |
| LocalStack FIFO ordering may not match real AWS | Accept as known limitation; FIFO tests validate message group ID propagation, not strict ordering |
| Delayed message support in SQS may behave differently than template expects | The template uses a 5-second delay; verify this works with SQS `DelaySeconds` attribute in LocalStack |
| V4 and non-V4 test code divergence | Both use identical `GatewayFactory`/`CredentialsChain` patterns; namespace is the only difference |
