# Plan: GCP Pub/Sub Generated Tests Migration

## Summary

Replace 20 manually-written GCP Pub/Sub messaging gateway tests with auto-generated tests using the existing Liquid template framework. This requires adding `MessagingGateways` config (Pull + Stream variants) to `test-configuration.json` and implementing two provider classes.

## Technical Context

- **Language:** C# / .NET
- **Test Framework:** xUnit
- **Generator:** `tools/Paramore.Brighter.Test.Generator` (Liquid templates)
- **Reference Implementations:** Redis (single variant), RMQ (multi-variant Classic + Quorum), Kafka (single variant)
- **GCP Test Project:** `tests/Paramore.Brighter.Gcp.Tests/`

## Implementation Phases

### Phase 1: Configuration & Providers

**Goal:** Create the provider classes and update config so the generator can produce GCP messaging tests.

**T001: Update test-configuration.json**
- Add `MessagingGateways` dictionary with `Pull` and `Stream` entries
- Preserve existing `Outboxes` config
- Reference: `tests/Paramore.Brighter.RMQ.Async.Tests/test-configuration.json` (multi-variant pattern)
- File: `tests/Paramore.Brighter.Gcp.Tests/test-configuration.json`

**T002: Create GcpPullMessageGatewayProvider**
- Implement `IAmAMessageGatewayProactorProvider` and `IAmAMessageGatewayReactorProvider`
- Use `GatewayFactory` for connection config
- Create subscriptions with `SubscriptionMode.Pull`
- Support DLQ via `DeadLetterPolicy`, broker validation, requeue, ordering keys
- Reference: `tests/Paramore.Brighter.RMQ.Async.Tests/MessagingGateway/RmqClassicMessageGatewayProvider.cs`
- Reference: existing manual test setup in `Pull/Proactor/When_posting_a_message_via_the_messaging_gateway_async.cs`
- File: `tests/Paramore.Brighter.Gcp.Tests/MessagingGateway/GcpPullMessageGatewayProvider.cs`

**T003: Create GcpStreamMessageGatewayProvider**
- Same as T002 but with `SubscriptionMode.Stream`
- Reference: existing manual test setup in `Stream/Proactor/When_posting_a_message_via_the_messaging_gateway_async.cs`
- File: `tests/Paramore.Brighter.Gcp.Tests/MessagingGateway/GcpStreamMessageGatewayProvider.cs`

**T004: Create GcpMessageAssertion (if needed)**
- Check if GCP messages need custom assertion logic vs the default `MessageAssertion`
- If GCP ordering keys or other headers need special assertion, create `GcpMessageAssertion.cs`
- Reference: `tests/Paramore.Brighter.RMQ.Async.Tests/MessagingGateway/RmqMessageAssertion.cs`
- File: `tests/Paramore.Brighter.Gcp.Tests/MessagingGateway/GcpMessageAssertion.cs` (conditional)

**Parallelism:** T002 + T003 can run in parallel after T001.

### Phase 2: Generate & Validate (MVP)

**Goal:** Run the generator and verify generated tests compile and pass.

**T005: Run generator**
- Execute: `cd tests/Paramore.Brighter.Gcp.Tests && dotnet run --project ../../tools/Paramore.Brighter.Test.Generator`
- Verify `MessagingGateway/Generated/` directory is created with Pull + Stream subdirectories
- Verify expected test count (~40 files)

**T006: Build verification**
- Run `dotnet build` on the GCP test project
- Fix any compilation errors in providers

**T007: Run generated tests**
- Execute generated tests against GCP Pub/Sub emulator
- Compare behavior with existing manual tests
- Fix any provider issues

**MVP Checkpoint:** Generated tests pass. Manual tests still exist as safety net.

### Phase 3: Cleanup

**Goal:** Remove manual tests and verify no regressions.

**T008: Delete manual messaging gateway tests**
- Delete `MessagingGateway/Pull/` directory (10 files)
- Delete `MessagingGateway/Stream/` directory (10 files)
- Keep `Helper/` and `TestDoubles/` (shared with outbox tests)

**T009: Final validation**
- Full test suite run to confirm no regressions
- Verify outbox generated tests still pass
- Verify `generate-test.sh` runs cleanly for all projects

## Risk Mitigations

| Risk | Mitigation |
|------|-----------|
| Emulator doesn't support DLQ | Test manually first; skip DLQ template if unsupported |
| Ordering key tests fail on emulator | Set `HasSupportToPartitionKey: false` if emulator lacks support |
| Stream cleanup differs from Pull | Implement mode-specific cleanup in each provider |
| Template expects behavior GCP doesn't support | Use capability flags to conditionally skip tests |
