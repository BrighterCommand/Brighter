Fallback
If a request cannot be completed by Handle(TRequest), implementing the Fallback(TRequest) method provides an alternate code path that can be used This allows for graceful degradation. Using the FallbackPolicyAttribute handler you can configure a policy to catch either all Exception's or just BrokenCircuitException that occur later in the pipeline, and then call the Fallback(TRequest) path. Note that the FallbackPolicyAttribute target handler might be 'beginning of chain' and need to pass through to actual handler that is end of chain. Because of this we need to call Fallback on the chain. Later step handlers don't know the context of failure so they cannot know if any operations they had, that could fail (such as DB access) were the cause of the failure chain being hit. Steps that don't know how to handle should pass through. Useful alternatives for Fallback are to try via the cache. Note that a Fallback handler implementation should not catch exceptions in the Fallback(TRequest) chain to avoid an infinite loop. Call Successor.Handle(TRequest) if having provided a Fallback you want the chain to return to the 'happy' path. Excerise caution here though as you do not know who generated the exception that caused the fallback chain. For this reason, the [FallbackPolicyHandler] puts the exception in the request context. When the FallbackPolicyAttribute is set on the Handle(TRequest) method of a derived class The FallbackPolicyHandler<(Of <(<'TRequest>)>)> will catch either all failures (backstop) or BrokenCircuitException depending on configuration and call the RequestHandler<(Of <(<'TRequest>)>)>'s Fallback(TRequest) method
